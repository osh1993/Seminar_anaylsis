# 슬라이드 해설 작성 규칙

## 목표
해당 분야 비전공 학습자도 이해할 수 있는 **쉽고 정확한** 한국어 해설. 쉬움과 정확함이 충돌하면 정확함을 택하고, 비유는 "비유하자면"으로 구분한다.

## 인용 표기 규칙 (가장 중요)
모든 해설 문장에는 근거 슬라이드 태그를 붙인다. 이 태그는 웹페이지에서 클릭 가능한 링크가 된다.

- 형식: `[D1-p23]` (doc1의 23번 슬라이드), 복수 근거는 `[D1-p23][D1-p24]`
- 현재 슬라이드를 설명하는 문장에도 자기 태그를 붙인다 — 웹에서 문장 단위 추적을 위해.
- 다른 슬라이드 내용을 끌어오면 그 슬라이드 태그를 쓴다.
- 슬라이드에 없는 배경지식 보충은 태그 대신 `(배경지식)` 마커를 붙인다. 태그와 마커가 둘 다 없는 문장은 금지.
- 존재하지 않는 페이지 태그는 QA에서 치명 결함으로 처리된다. 유효 범위는 `01_extract_day{N}.json`의 슬라이드 수 기준.

## 해설 품질 기준
- 전문용어는 첫 등장 시 한 줄로 풀어쓴다 (예: "프루닝(pruning, 가지치기: 중요도 낮은 연결을 잘라내 모델을 줄이는 기법)").
- 수식 슬라이드는 수식의 "의미"를 말로 푼다. 기호 나열 금지.
- 목차/구분/감사 슬라이드는 one_liner만 쓰고 explanation은 1문장으로 짧게.
- 그림 중심 슬라이드(텍스트 200자 미만)는 반드시 PNG 이미지를 읽고 그림이 말하는 바를 해설한다.

## 출력 스키마 (해설 모드)
파일: `{폴더}/_workspace/02_interpret_day{doc}_{start:03d}-{end:03d}.json`

```json
{
  "day": 1, "range": [1, 36],
  "slides": [{
    "day": 1, "page": 5,
    "section": "Background",
    "title": "슬라이드 제목(원문)",
    "one_liner": "한 줄 요약",
    "explanation": "쉬운 해설 2~6문장. 각 문장 끝에 [D1-p5] 태그 또는 (배경지식) 마커.",
    "key_terms": [{"term": "Pruning", "easy": "한 줄 쉬운 정의"}],
    "deep_dive_candidates": ["Pruning"],
    "difficulty": "basic | intermediate | advanced",
    "references": [{"url": "슬라이드 원문에 있는 URL 그대로", "context": "무엇의 출처인지"}],
    "related_slides": [{"day": 1, "page": 51, "why": "SVD 상세"}],
    "qa": [{"q": "예상 질문", "a": "답변. 문장마다 인용 태그 필수."}]
  }]
}
```

- `section`: 슬라이드에 브레드크럼/장 표기가 있으면 그것을 쓰고, 없으면 제목 흐름으로 단원을 판단해 일관된 섹션명을 부여한다.
- `key_terms`: 슬라이드에 실제 등장한 용어만. 구간 내 중복 등장 시 첫 슬라이드에만.
- `deep_dive_candidates`: 이 세미나의 뼈대가 되는 개념(구간당 3~8개 수준)만 표시 — concept-researcher의 개념 사전 후보가 된다. 모든 key_term을 넣지 않는다.
- `qa`: 핵심 슬라이드에 1~2개, 사소한 슬라이드는 빈 배열 허용. 웹페이지 Q&A 기능의 데이터가 된다.
- `references.url`은 입력 JSON의 `urls` 배열 또는 슬라이드 원문에 있는 것만 사용한다. **URL 생성 금지.**

## 작업 순서 (권장)
1. 담당 구간의 입력 JSON을 읽고 전체 흐름을 파악한다.
2. 텍스트 빈약 슬라이드 목록을 만들어 해당 PNG를 읽는다.
3. 슬라이드 순서대로 해설을 작성하고 한 파일로 저장한다.
4. 저장 전 자기 검사: 태그 페이지 범위, URL이 원문에 있는지, 스키마 유효성.

# 강의 실황(lecture_note) 오버레이 작성 규칙

강의 녹취록(transcript)이 제공될 때만 실행한다. 슬라이드 해설(explanation)은 **슬라이드에 적힌 내용**을, lecture_note는 강사가 **말로만 전달한** 직관·이유·업계 사례·강의 중 문답을 담는다. 별도 오버레이 파일로 분리해 웹에서 "🎙️ 강의 실황" 블록으로 렌더한다.

## 작성 원칙
- **중복 금지**: 슬라이드에 적힌 내용 반복 금지. 강의에서만 나온 맥락·이유·비유·실무 팁·사례만.
- **인용 태그 동일 규칙**: 모든 문장에 `[D1-p32]` 태그 또는 `(배경지식)` 마커.
- **불명확 표시 보존**: 녹취록에 `[불명확]` 표기된 고유명사는 단정하지 말고 "~로 언급됨/추정"으로 전달한다.
- 대응 서사가 없는 슬라이드는 건너뛴다. 분량은 슬라이드당 2~5문장.
- 실제 강의 문답만 `lecture_qa`에 넣는다 (예상 질문과 구분).

## 오버레이 스키마
파일: `_workspace/05_lecture_day{doc}_{start:03d}-{end:03d}.json`
```json
{"day": 1, "range": [1, 44], "source_transcript": ["파일명.md"],
 "notes": [{"page": 32, "lecture_note": "문장마다 태그 [D1-p32].",
            "lecture_qa": [{"q": "수강생 질문", "a": "강사 답변. 태그 필수."}]}]}
```

# 종합 큐레이션 모드 (강조 / 연구 흐름)

전체 해설 산출물을 읽고 세미나를 관통하는 두 종합 뷰 데이터를 만든다. 학습자가 "큰 흐름"을 먼저 잡는 용도다.

## ⭐ 강조 (highlights)
파일: `_workspace/06_highlights_day{doc}.json` — 카테고리(핵심 기법/실무 조언/주의·함정/강사 평가/산업 인사이트)별 중요 항목 20~35개.
```json
{"day": 1, "items": [{"category": "핵심 기법", "importance": "high|mid",
  "title": "한 줄 제목", "note": "왜 중요한가 1~3문장. 태그 필수.",
  "pages": [{"day":1,"page":32}]}]}
```

## 🧭 연구 흐름 (flow)
파일: `_workspace/07_flow_day{doc}.json` — 세미나 내용을 주제 갈래(track)로 나누고, 각 갈래를 시간/논리 순 타임라인으로 구성한다.
```json
{"day": 1, "intro": "이 세미나의 전체 지도를 2~3문장으로.",
 "tracks": [{"name": "갈래명", "steps": [{
   "label": "기법/개념명", "era": "시기(알 때만)",
   "concept": "한 줄 개념 (인용 태그 없는 요약 레이블)",
   "transition": "이전 기법의 한계 → 이 기법이 등장한 이유 (다음 step으로 가는 다리)",
   "note": "상세 설명. 문장마다 인용 태그.", "pages": [{"day":1,"page":51}]}]}]}
```
- `concept`/`transition`은 요약 레이블이라 인용 태그 생략 가능 — 근거는 note·pages가 담당.
- 갈래 수는 자료 성격에 맞게 (기술 세미나 5~10개, 방법론 세미나 3~6개 수준).
