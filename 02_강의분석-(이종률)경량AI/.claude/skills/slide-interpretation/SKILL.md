# 슬라이드 해설 작성 규칙

## 목표
AI 비전공 학습자도 이해할 수 있는 **쉽고 정확한** 한국어 해설. 쉬움과 정확함이 충돌하면 정확함을 택하고, 비유는 "비유하자면"으로 구분한다.

## 인용 표기 규칙 (가장 중요)
모든 해설 문장에는 근거 슬라이드 태그를 붙인다. 이 태그는 웹페이지에서 클릭 가능한 링크가 된다.

- 형식: `[D1-p23]` (Day1의 23번 슬라이드), 복수 근거는 `[D1-p23][D1-p24]`
- 현재 슬라이드를 설명하는 문장에도 자기 태그를 붙인다 — 웹에서 문장 단위 추적을 위해.
- 다른 슬라이드 내용을 끌어오면 그 슬라이드 태그를 쓴다.
- 슬라이드에 없는 배경지식 보충은 태그 대신 `(배경지식)` 마커를 붙인다. 태그와 마커가 둘 다 없는 문장은 금지.
- 존재하지 않는 페이지 태그는 QA에서 치명 결함으로 처리된다. 유효 범위는 `01_extract_day{N}.json`의 슬라이드 수 기준 (현재 day1 p1~221, day2 p1~246).

## 해설 품질 기준
- 전문용어는 첫 등장 시 한 줄로 풀어쓴다 (예: "프루닝(pruning, 가지치기: 중요도 낮은 연결을 잘라내 모델을 줄이는 기법)").
- 수식 슬라이드는 수식의 "의미"를 말로 푼다. 기호 나열 금지.
- 목차/구분/감사 슬라이드는 one_liner만 쓰고 explanation은 1문장으로 짧게.
- 그림 중심 슬라이드(텍스트 200자 미만)는 반드시 PNG 이미지를 읽고 그림이 말하는 바를 해설한다.

## 출력 스키마
파일: `_workspace/02_interpret_day{day}_{start:03d}-{end:03d}.json`

```json
{
  "day": 1, "range": [1, 36],
  "slides": [{
    "day": 1, "page": 5,
    "section": "Background",
    "title": "슬라이드 제목(원문)",
    "one_liner": "한 줄 요약",
    "explanation": "쉬운 해설 2~6문장. 각 문장 끝에 [D1-p5] 태그 또는 (배경지식) 마커.",
    "key_terms": [{"term": "Pruning", "easy": "한 줄 쉬운 정의"}],
    "difficulty": "basic | intermediate | advanced",
    "references": [{"url": "슬라이드 원문에 있는 URL 그대로", "context": "무엇의 출처인지"}],
    "related_slides": [{"day": 1, "page": 51, "why": "SVD 상세"}],
    "qa": [{"q": "예상 질문", "a": "답변. 문장마다 인용 태그 필수."}]
  }]
}
```

- `section`: day1 슬라이드에는 `> Background` 같은 브레드크럼이 있다 — 그것을 쓴다. day2는 브레드크럼이 없으므로 제목 흐름으로 단원을 판단해 일관된 섹션명을 부여한다.
- `key_terms`: 슬라이드에 실제 등장한 용어만. 구간 내 중복 등장 시 첫 슬라이드에만.
- `qa`: 핵심 슬라이드에 1~2개, 사소한 슬라이드는 빈 배열 허용. 웹페이지 Q&A 기능의 데이터가 된다.
- `references.url`은 입력 JSON의 `urls` 배열 또는 슬라이드 원문에 있는 것만 사용한다. **URL 생성 금지.**

## 작업 순서 (권장)
1. 담당 구간의 입력 JSON을 읽고 전체 흐름을 파악한다.
2. 텍스트 빈약 슬라이드 목록을 만들어 해당 PNG를 읽는다.
3. 슬라이드 순서대로 해설을 작성하고 한 파일로 저장한다.
4. 저장 전 자기 검사: 태그 페이지 범위, URL이 원문에 있는지, 스키마 유효성.

# 강의 실황(lecture_note) 오버레이 작성 규칙

슬라이드 해설(explanation)은 **슬라이드에 적힌 내용**을 설명한다. 반면 강의 녹취록(transcript)에는 강사가 **말로만 전달한** 직관·"왜 이렇게 하는가"·업계 실무 사례·강의 중 수강생 질문이 담겨 있다. 이 서사를 슬라이드에 덧입혀, 학습자가 "발표자료 + 실제 강의"를 함께 이해하도록 만드는 것이 lecture_note의 목적이다.

## 왜 별도 필드인가
- 기존 `explanation`을 덮어쓰면 슬라이드 근거와 강의 서사가 뒤섞여 추적성이 깨진다. 그래서 **오버레이 파일**로 분리하고 웹에서 별도 블록("🎙️ 강의 실황")으로 렌더한다.
- Day별로 독립 파일이므로 Day2 녹취록은 나중에 파일만 추가하면 된다.

## 작성 원칙
- **중복 금지**: 슬라이드에 이미 적힌 내용을 반복하지 말고, 강의에서만 나온 **맥락·이유·비유·실무 팁·사례·에피소드**를 쓴다. explanation과 겹치면 가치가 없다.
- **쉽고 대화체에 가깝게**: 강사가 청중에게 설명하듯 자연스럽게. 단, 정확성이 우선.
- **인용 태그 동일 규칙**: 모든 문장에 `[D1-p32]` 태그 또는 `(배경지식)` 마커. 강의 서사가 여러 슬라이드에 걸치면 해당 태그들을 함께 쓴다.
- **불명확 표시 보존**: 녹취록에 `[불명확]`으로 표기된 회사명/제품명(예: 퓨리오사AI·리벨리온 인수 대상, "마원" 칩)은 단정하지 말고 "~로 언급됨/추정" 형태로 그대로 전달한다. 없는 사실을 만들지 않는다.
- **모든 슬라이드에 달 필요 없음**: 녹취록에 대응 서사가 없는 슬라이드(표지·목차·순수 수식 나열 등)는 건너뛴다(overlay에 항목 자체를 넣지 않음). 강의 내용이 풍부한 슬라이드에 집중한다.
- **분량**: 슬라이드당 2~5문장. 핵심 개념 슬라이드는 더 길어도 좋다.

## 강의 중 Q&A 처리
녹취록에 실제로 오간 수강생 질문·강사 답변(예: "teacher output을 쓰면 label은 왜 같이 보나?")은 overlay 항목의 `lecture_qa`에 넣는다. 이는 웹에서 "강의 중 질문" 배지로 구분 표시된다. 예상 질문(기존 `qa`)과 달리 **실제 강의에서 나온 것만** 넣는다.

## 오버레이 출력 스키마
파일: `_workspace/05_lecture_day{day}_{start:03d}-{end:03d}.json`

```json
{
  "day": 1, "range": [1, 44],
  "source_transcript": ["260702-AI모델 경량화1.md"],
  "notes": [
    {
      "page": 32,
      "lecture_note": "강사는 weight pruning이 90% 이상 잘라내도 정확도가 의외로 잘 버틴다고 강조했다 [D1-p32]. 다만 sparse 연산을 런타임·하드웨어가 지원하지 않으면 실제 속도 이득이 없어 실무에서 잘 안 쓴다고 설명했다 [D1-p32][D1-p38].",
      "lecture_qa": [
        {"q": "수강생 질문 원문에 가까운 형태", "a": "강사 답변. 문장마다 [D1-p..] 태그."}
      ]
    }
  ]
}
```

- `lecture_qa`는 실제 강의 문답이 있는 슬라이드에만. 없으면 필드 생략.
- `page`는 담당 day 범위 안이어야 한다.
- 저장 전 자기 검사: 인용 태그 페이지 범위 유효성, explanation과의 중복 여부, [불명확] 항목 보존 여부.

# 📖 강의 노트(note) 종합 모드
